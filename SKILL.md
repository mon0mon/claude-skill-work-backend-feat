# work-backend-feat — 백엔드 기능 구현 워크플로우

## 핵심 철학

이 스킬은 **"구조화된 워크플로우로 품질과 효율을 동시에 달성한다"**는 원칙 위에 설계되었다. 각 단계마다 AI 리뷰가 사용자 리뷰 전에 선행되어, 사용자가 핵심 판단에만 집중할 수 있도록 한다.

**사용자 승인 규칙**: 모든 승인 요청은 `(Y/n)` 형식으로 한다. 공백이나 빈 응답은 **긍정(Yes)**으로 처리한다.

---

## Superpowers 스킬 안내

이 워크플로우는 아래 superpowers 스킬이 설치되어 있으면 각 단계에서 자동으로 활용하여 더 높은 품질의 결과를 제공한다. 설치되어 있지 않아도 독립적으로 동작하지만, 설치를 권장한다.

**활용 스킬 목록**:
| 스킬 | 활용 단계 | 역할 |
|------|----------|------|
| `spec-reviewer` | Phase 2 (스펙 리뷰) | 병렬 전문가 관점 리뷰 |
| `test-driven-development` | Phase 6 (TDD) | TDD 프로세스 가이드 |
| `requesting-code-review` | Phase 6 (코드 리뷰) | 구조화된 코드 리뷰 |
| `writing-plans` | Phase 4 (실행 계획) | 체계적 실행 계획 수립 |

**설치 방법**: superpowers 스킬은 Claude Code 설정에서 관리된다. 스킬이 없는 경우 이 워크플로우 내 독립 로직으로 동일한 단계를 수행한다.

## 진행 상황 표시

모든 단계의 **시작 시와 끝(사용자 응답을 기다리는 지점 직전)** 에 아래 형식으로 현재 진행 상황을 표시한다. 사용자가 전체 워크플로우에서 어디에 있는지, 앞으로 얼마나 남았는지 항상 파악할 수 있어야 한다. 출력이 길어져 상단 표시가 스크롤 밖으로 밀려날 수 있으므로, **사용자에게 승인/선택을 요청하는 직전에도 반드시 진행 상황을 다시 표시한다.**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 Phase {N}/9: {Phase 이름}  |  Step {M}: {Step 이름}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[■■■■■□□□□] 56% (Phase 5/9)

Phase 1 ✅ → Phase 2 ✅ → Phase 3 ✅ → 【Phase 4】→ Phase 5 → ... → Phase 9
```

**리뷰 반복 시**: 리뷰가 여러 차례 반복되는 경우 반복 횟수와 남은 예상을 표시한다:
```
🔄 리뷰 반복: 2/3회차 (최대 3회)
- 1회차: CRITICAL 2건, HIGH 3건 → 피드백 반영 완료
- 2회차(현재): HIGH 1건 남음
- 예상: 이번 회차에서 수렴 가능
```

## 워크플로우 재개 (이전 대화 이어가기)

사용자가 기존 작업을 이어가려는 경우 (예: "PS-85 이어서 진행해줘"):

1. `docs/workflow/work/backend/spec/` 및 `docs/workflow/work/backend/plans/` 디렉토리에서 해당 티켓의 문서를 검색한다
2. 문서가 존재하면 읽어서 현재 상태를 파악한다
3. git 브랜치 상태를 확인하여 어느 단계까지 진행되었는지 판단한다
4. 해당 단계부터 워크플로우를 재개한다

---

## Phase 1: 요구사항 수집 & 명세 확인

### Step 1.1: 작업 내용 파악

사용자의 프롬프트에서 작업 내용을 파악한다. 작업 유형을 식별한다:
- **기능 구현**: 새로운 도메인/API/엔드포인트 추가
- **버그 수정**: 기존 기능의 결함 수정

### Step 1.2: 명세 확인 질문

설계에 앞서 아래 관점에서 불명확한 부분이 있으면 사용자에게 질문한다:

- **비즈니스 로직**: 요구사항이 모호하거나 여러 해석이 가능한 부분
- **설계 판단**: 아키텍처/패턴 선택이 필요한 부분
- **잠재적 문제**: 기존 코드와의 충돌, 성능 이슈, 데이터 정합성 문제 가능성
- **영향 범위**: 변경이 미치는 다른 도메인/모듈의 범위

질문 형식은 AI 리뷰 형식을 따른다 (Phase 3 참조).

정보가 충분하면 질문 없이 다음 단계로 진행한다.

---

## Phase 2: 스펙 문서 작성

### Step 2.0: Jira 프로젝트 키 확인

스펙 문서 파일명에 Jira 프로젝트 키(`{SPACE}`)가 필요하므로, 문서 작성 전에 프로젝트 키를 확정한다.

1. **프로젝트 CLAUDE.md에 `jira_project_key`가 명시되어 있으면** → 해당 키를 사용하고 질문 없이 진행
2. **메모리에 기록되어 있으면** → 해당 키를 사용하고 질문 없이 진행
3. **위 둘 다 없으면** → 사용자에게 질문:
   ```
   이 작업에 사용할 Jira 프로젝트 키를 알려주세요. (예: PS, TR, FE 등)
   ```
4. 사용자가 프로젝트 키를 알려주면, **프로젝트 CLAUDE.md에 자동으로 저장**한다:
   ```markdown
   ## Jira
   - Project Key: {SPACE}
   ```
   저장 후 안내:
   ```
   Jira 프로젝트 키 '{SPACE}'를 CLAUDE.md에 저장했습니다. 다음부터는 자동으로 사용합니다.
   ```

### Step 2.1: 스펙 문서 생성

`references/spec-template.md`의 템플릿을 참조하여 스펙 문서를 작성한다.

**저장 경로**: `docs/workflow/work/backend/spec/{SPACE}-XX_{YYMMDD}_spec_{타이틀}.md`
- 아직 Jira 티켓이 없으므로 `{SPACE}-XX`는 placeholder로 사용한다. `{SPACE}`는 Step 2.0에서 확정한 Jira 프로젝트 키이다
- `{YYMMDD}`는 문서 생성 시점의 날짜이다 (예: `260318`). 동일 타이틀의 파일명 충돌을 방지한다
- 타이틀은 기능명을 kebab-case로 작성한다 (예: `PS-XX_260318_spec_user-profile-image.md`)

### Step 2.2: 스펙 문서 AI 리뷰

스펙 문서를 사용자에게 보여주기 전에 AI 리뷰를 수행한다.

**기존 스킬 활용**: `spec-reviewer` 스킬이 사용 가능하면 해당 스킬을 호출한다. 사용 불가능하면 아래 독립 리뷰를 수행한다.

독립 리뷰 시 Phase 3의 AI 리뷰 형식에 따라 다음 관점에서 검토한다:
- 요구사항 누락 여부
- 도메인 모델 설계의 적절성
- API 설계의 일관성 (기존 프로젝트 컨벤션과 비교)
- 영향 범위의 정확성

### Step 2.3: 사용자 리뷰

AI 리뷰 결과와 함께 스펙 문서를 사용자에게 제시한다.

- 통과: Phase 4로 진행
- 미통과: 피드백을 반영하여 Phase 1의 질문 단계부터 재수행

## Phase 3: AI 리뷰 형식

모든 단계의 AI 리뷰에서 발견된 문제는 다음 형식으로 정리한다. 이 형식은 사용자가 빠르게 판단하고 선택할 수 있도록 구조화되어 있다.

### 문제 정의 형식

각 문제는 아래 구조를 따른다:

```
### [P{번호}] {문제 제목}  [CRITICAL | HIGH | MEDIUM | LOW]

**도메인/상황**: {이 문제가 발생하는 도메인이나 상황, 유즈케이스에 대한 자세한 설명}

**문제 설명**: {구체적으로 어떤 문제인지 상세 설명}

**발생 시나리오**: {문제가 발생하는 특정 유즈케이스나 상황, 그리고 그 결과가 어떻게 되는지 자세히 설명}

**해결 방안**:

  A) {해결 방법 설명}
     - 근거: {왜 이 방법을 제안하는지}
     - 장점: {이 방법의 장점}
     - 단점: {이 방법의 단점}
     - 현재 상황 적합도:
       | 관점 | 적합도 | 설명 |
       |------|--------|------|
       | 프로젝트 아키텍처 | ✅ 적합 / ⚠️ 보통 / ❌ 부적합 | {근거} |
       | 기존 코드 일관성 | ✅ / ⚠️ / ❌ | {근거} |
       | 구현 복잡도 | ✅ 낮음 / ⚠️ 보통 / ❌ 높음 | {근거} |
       | 유지보수성 | ✅ / ⚠️ / ❌ | {근거} |
       | 성능 영향 | ✅ 없음 / ⚠️ 경미 / ❌ 유의미 | {근거} |
       | 테스트 용이성 | ✅ / ⚠️ / ❌ | {근거} |
     - 추천 여부: {추천 / 비추천 / 조건부 추천}

  B) {해결 방법 설명}
     - 근거: ...
     - 장점: ...
     - 단점: ...
     - 현재 상황 적합도: (동일 테이블 형식)
     - 추천 여부: ...

  C) {해결 방법 설명}
     - ...

  👉 추천: {A/B/C} — {추천 이유 한 줄}

  📖 확장 설명이 필요하면 `P{번호}+` 를 입력하세요.
```

### 확장 설명 형식

사용자가 특정 문제에 대해 `P{번호}+`를 입력하면 해당 문제의 확장 설명을 제공한다. 확장 설명은 일반 문제 정의보다 훨씬 더 깊이 있고 디테일한 분석을 제공한다:

```
### [P{번호}] {문제 제목} — 확장 분석

**상세 배경**:
{문제가 발생하게 된 기술적/비즈니스적 맥락을 풍부하게 서술.
관련 도메인 지식, 유사한 문제가 발생하는 패턴, 이 프로젝트에서의 특수성 등을 포함.
필요시 코드 예시나 시퀀스 다이어그램을 포함한다.}

**심층 시나리오 분석**:
{단순히 "이런 상황에서 문제가 발생한다"가 아니라, 구체적인 데이터 흐름과
사용자 행동 시퀀스를 추적하면서 어디서 어떻게 문제가 표면화되는지 단계별로 설명.
엣지 케이스와 경계 조건도 포함한다.}

**해결 방안 심층 비교**:

  A) {해결 방법}
     - 상세 근거: {기술적 근거를 깊이 있게 설명. 관련 프레임워크 동작 원리,
       비슷한 상황에서의 업계 표준 접근법, 이 접근법의 이론적 배경 등}
     - 장점 상세:
       - {장점 1의 구체적 효과와 예시}
       - {장점 2의 구체적 효과와 예시}
     - 단점 상세:
       - {단점 1이 실제로 발생하는 조건과 영향 정도}
       - {단점 2와 이를 완화할 수 있는 보조 전략}
     - 현재 상황 적합도: (테이블 형식 — 일반 형식과 동일하되 설명을 더 상세히)
     - 구현 예시: (필요시 코드 스니펫)
     - 추천 여부: {추천 근거를 2~3문장으로}

  B) ...
  C) ...

**관련 사례 / 참고**:
{비슷한 문제에 대한 일반적인 해결 패턴이나, 이 프로젝트 내 유사 구현 사례 참조}
```

### 리뷰 요약 섹션

리뷰의 마지막에는 반드시 다음 형식의 요약을 추가한다:

```
---

## 리뷰 요약

| # | 문제 | 심각도 | 추천 해결방안 | 핵심 판단 포인트 |
|---|------|--------|-------------|----------------|
| P1 | {제목} | CRITICAL | {추천안 레이블} | {사용자가 판단해야 할 핵심} |
| P2 | {제목} | HIGH | {추천안 레이블} | {핵심 판단 포인트} |
| P3 | {제목} | MEDIUM | {추천안 레이블} | {핵심 판단 포인트} |
| P4 | {제목} | LOW | {추천안 레이블} | {핵심 판단 포인트} |

**즉시 결정 필요**: P1 (CRITICAL)
**검토 권장**: P2, P3 (HIGH/MEDIUM)
**참고 사항**: P4 (LOW)
**발견된 문제 없음인 영역**: {문제가 없었던 관점 나열}

💡 각 문제에 대해 `P1: A, P2: B` 식으로 선택하세요.
📖 상세 분석이 필요하면 `P{번호}+` 를 입력하세요.
```

## Phase 4: 실행 계획 작성

### Step 4.1: 실행 계획 문서 생성

`references/plan-template.md`의 템플릿을 참조하여 실행 계획을 작성한다.

**저장 경로**: `docs/workflow/work/backend/plans/{SPACE}-XX_{YYMMDD}_plan_{타이틀}.md`

실행 계획에는 TDD 접근법을 반영한다:
1. 각 단계에서 먼저 작성할 테스트 목록
2. 테스트를 통과시키기 위한 구현 작업
3. 커밋 단위 (모듈/유즈케이스 단위, 작고 가독성 있는 규모)
4. **API 엔드포인트가 추가/변경되는 경우, 반드시 마지막 Step으로 JetBrains HTTP Client(`.http` 파일) 업데이트를 포함한다** (Step 6.2.1 참조)

### Step 4.2: 실행 계획 AI 리뷰

Phase 3의 AI 리뷰 형식에 따라 실행 계획을 검토한다:

### Step 4.3: 사용자 리뷰 & 실행 확인

### Step 6.2.1: JetBrains HTTP Client (.http 파일) 업데이트

API 엔드포인트가 추가되거나 변경된 경우, `http/` 디렉토리의 JetBrains HTTP Client 파일도 함께 업데이트한다. 이 파일들은 개발자가 IntelliJ에서 API를 빠르게 테스트할 수 있도록 하는 스크립트이다.

**동작 규칙**:
1. 구현된 API 엔드포인트를 확인한다
2. `http/` 디렉토리에서 해당 도메인의 `.http` 파일을 찾는다
   - 기존 파일이 있으면 해당 파일에 추가/수정
   - 없으면 새 `.http` 파일을 생성 (도메인명 기반, 예: `check-in.http`)
3. 기존 `.http` 파일의 스타일과 형식을 따른다

**`.http` 파일 작성 컨벤션** (기존 프로젝트 패턴 참조):
```
@variable = defaultValue

### {API 설명}
{METHOD} {{host}}/{path}
Authorization: {{jwtToken}}
Content-Type: application/json

{요청 바디 (POST/PUT인 경우)}
```

- 환경 변수는 `{{host}}`, `{{jwtToken}}` 등 `http-client.env.json`에 정의된 것을 사용
- 요청 파라미터는 파일 상단에 `@변수 = 기본값` 형태로 선언
- 각 요청은 `###`로 구분하고 한글 설명을 붙인다
- 인증이 필요한 엔드포인트는 `Authorization: {{jwtToken}}` 헤더 포함
- 로그인 등 토큰을 반환하는 엔드포인트는 JavaScript 응답 핸들러로 토큰 자동 저장 포함

### Step 6.3: 구현 코드 AI 리뷰

구현 완료 후, Phase 3의 AI 리뷰 형식에 따라 코드를 검토한다. 아래의 모든 관점을 빠짐없이 체크한다.

**기존 스킬 활용**: `requesting-code-review` 스킬이 사용 가능하면 활용한다.

#### 6.3.1 스펙 정합성
- 스펙 문서의 모든 요구사항이 구현되었는지 확인
- 스펙에 없는 기능이 추가되지 않았는지 확인

#### 6.3.2 아키텍처 컨벤션 준수
다음 아키텍처 규칙 위반 여부를 체크한다:
- **헥사고날 아키텍처 위반**: api 모듈이 common의 repository를 직접 사용하는지, common 모듈 간 유즈케이스 호출이 있는지 (밀접하지 않은 도메인 간)
- **계층 침범**: 컨트롤러에서 비즈니스 로직 수행, 서비스에서 HTTP 관련 로직 처리
- **패키지 구조 위반**: 파일이 컨벤션에 맞지 않는 위치에 있는지
- **네이밍 컨벤션 위반**: 포트 인터페이스, 컨트롤러, DTO 등의 이름이 프로젝트 규칙을 따르는지
- **의존 방향 위반**: 내부 레이어가 외부 레이어에 의존하는지

#### 6.3.3 JPA & 데이터베이스 문제
다음 패턴을 반드시 확인한다:

| 체크 항목 | 설명 | 발생 조건 |
|----------|------|----------|
| **N+1 쿼리** | 연관 엔티티를 루프에서 개별 조회 | `@OneToMany`/`@ManyToOne` 관계에서 Lazy 로딩 시 컬렉션 순회 |
| **MultiBagFetchException** | 2개 이상의 `@OneToMany` 관계를 동시에 fetch join | `List` 타입 컬렉션 2개 이상 fetch join |
| **Pagination In-Memory** | 페이징 쿼리가 모든 데이터를 메모리에 로드 후 페이징 | fetch join + Pageable 조합 시 |
| **Detached Entity Dirty Checking** | 트랜잭션 밖에서 엔티티 수정 후 dirty checking 기대 | `@Transactional` 범위 밖에서 setter 호출 |
| **LazyInitializationException** | 트랜잭션 밖에서 Lazy 로딩 프록시 접근 | 컨트롤러/뷰 레이어에서 미로딩 연관 엔티티 접근 |
| **영속성 컨텍스트 혼동** | 같은 엔티티의 다른 인스턴스 공존 | 서로 다른 트랜잭션/영속성 컨텍스트에서 동일 ID 엔티티 조작 |
| **CascadeType 오용** | 의도치 않은 연쇄 삭제/저장 | `CascadeType.ALL`이나 `REMOVE` 사용 시 |
| **orphanRemoval 부작용** | 컬렉션 교체 시 의도치 않은 삭제 | `orphanRemoval = true`와 컬렉션 재할당 |

#### 6.3.4 동시성 & 트랜잭션 문제

| 체크 항목 | 설명 | 발생 조건 |
|----------|------|----------|
| **Lost Update** | 동시 수정 시 나중 트랜잭션이 앞선 변경 덮어씀 | 락 없이 read-modify-write 패턴 |
| **Race Condition** | 동시 요청으로 데이터 불일치 | 중복 등록, 동시 좋아요/취소, 잔액 차감 등 |
| **Deadlock** | 두 트랜잭션이 서로의 락 대기 | 여러 테이블을 다른 순서로 락 획득 |
| **잘못된 락 사용** | 불필요한 비관적 락으로 성능 저하 | `@Lock(PESSIMISTIC_WRITE)` 과다 사용 |
| **트랜잭션 범위 과대** | 긴 트랜잭션이 DB 커넥션 점유 | 외부 API 호출을 트랜잭션 안에서 수행 |
| **비동기 내 동기 블로킹** | 비동기 컨텍스트에서 동기 코드 실행 | `@Async` 메서드 안에서 동기적 DB 조회 반복 |

#### 6.3.5 성능 문제

| 체크 항목 | 설명 | 발생 조건 |
|----------|------|----------|
| **대량 데이터 풀스캔** | 인덱스 없이 전체 테이블 스캔 | WHERE 절에 인덱스 미적용 컬럼 사용 |
| **불필요한 전체 컬럼 조회** | SELECT * 패턴 | 엔티티 전체를 조회하나 일부 필드만 사용 |
| **메모리 누수 패턴** | 큰 컬렉션을 메모리에 한번에 로드 | `findAll()` 후 스트림 필터링 |
| **커넥션 풀 고갈** | 장시간 커넥션 점유 | 트랜잭션 안에서 외부 API 호출, 파일 I/O |
| **캐시 미활용** | 반복 조회되는 데이터에 캐시 미적용 | 자주 조회되나 변경 빈도 낮은 데이터 |
| **비동기 오버헤드** | 단순 작업에 비동기 처리 | 작은 작업에 `@Async` 적용으로 스레드 전환 비용 발생 |

#### 6.3.6 일반적인 백엔드 실수

| 체크 항목 | 설명 |
|----------|------|
| **@Transactional(readOnly=true) 누락** | 읽기 전용 메서드에 readOnly 미설정 |
| **예외 처리 누락** | 외부 API 호출, 파일 I/O 등에서 예외 미처리 |
| **입력 검증 누락** | `@field:` 접두어 없는 Bean Validation, 서비스 레이어 검증 누락 |
| **Enum 매핑 오류** | `@Enumerated(EnumType.ORDINAL)` 사용 (STRING이어야 함) |
| **FetchType.EAGER 사용** | `@ManyToOne`/`@OneToMany`에서 EAGER 로딩 |
| **equals/hashCode 미구현** | 엔티티 비교 시 ID 기반 비교 미구현 |
| **날짜/시간 처리 오류** | 타임존 미고려, `LocalDateTime` vs `ZonedDateTime` 혼용 |
| **민감 정보 노출** | 응답 DTO에 비밀번호, 내부 ID 등 노출 |
| **findByIdOrThrow 미사용** | `findById().orElseThrow()` 대신 프로젝트 확장 함수 미사용 |

#### 6.3.7 테스트 커버리지
- 실행 계획의 모든 테스트가 작성되었는지
- 테스트가 의미 있는 검증을 수행하는지 (단순 not null 체크가 아닌)
- 테스트 픽스처가 `InstancioUtil.kt`에 추가되었는지

- 본문: 어떤 사항을 변경했는지 **자세히** 기술한다. 추가/수정/삭제된 파일과 그 목적을 항목별로 나열한다
- 맥락이 필요한 경우: 본문 마지막에 짧은 요약(왜 이 변경이 필요했는지, 어떤 배경이 있는지)을 추가한다

```
PS-85 뉴스 댓글 등록 API 및 도메인 로직 구현

- NewsComment 엔티티 추가 (BaseAggregateRoot 상속, News/User ManyToOne 관계)
- RegisterNewsComment 포트 인터페이스 및 Command 추가
- NewsCommentService에 댓글 등록 로직 구현 (@Transactional)
- NewsCommentRepository JpaRepository 추가
- RegisterNewsCommentTest 유즈케이스 테스트 작성

뉴스 상세 페이지에 댓글 기능이 추가됨에 따라 댓글 등록 도메인 레이어를 먼저 구현
```

### Step 9.1: 머지 대기

```

```

### Step 9.2: 머지 실행

승인 시 PR을 머지한다.

### Step 9.3: 브랜치 정리

머지 완료 후 로컬과 원격 브랜치를 삭제한다.

1. `develop` 브랜치로 체크아웃: `git checkout develop`
2. 최신 상태 동기화: `git pull origin develop`
3. 로컬 브랜치 삭제: `git branch -d {브랜치명}`
4. 원격 브랜치 삭제: `git push origin --delete {브랜치명}`
