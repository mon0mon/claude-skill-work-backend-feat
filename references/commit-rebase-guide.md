# 커밋 정리 & Rebase 가이드

Push 전에 로컬 커밋들을 정리하고 base 브랜치와 동기화하는 절차이다.

## 1. 커밋 컨벤션 감지

커밋 메시지 컨벤션은 프로젝트마다 다르므로, 다음 순서로 감지한다:

1. **프로젝트 CLAUDE.md에 커밋 컨벤션이 명시되어 있으면** → 해당 규칙을 따른다
2. **프로젝트에 커밋 관련 설정 파일이 있으면** → 해당 규칙을 파싱한다
   - `.commitlintrc`, `.commitlintrc.json`, `.commitlintrc.yml`
   - `commitlint.config.js`, `commitlint.config.ts`
   - `.czrc`, `.cz.json` (commitizen)
3. **위 둘 다 없으면** → `git log --oneline -20`으로 최근 커밋 20개를 분석하여 패턴을 추출한다
   - 접두어 패턴 (예: `PS-XX`, `feat:`, `fix:`, `[PROJ-123]`)
   - 제목 줄 길이, 본문 사용 여부
   - 언어 (한국어/영어)

**감지 결과를 사용자에게 보여주고 확인받는다**:
```
커밋 컨벤션을 감지했습니다:
- 형식: `{SPACE}-{번호} {한국어 설명}`
- 본문: 변경사항 항목별 나열
- 예시: `PS-85 뉴스 댓글 등록 API 및 도메인 로직 구현`

이 컨벤션으로 커밋을 정리할까요? (Y/n)
```

## 2. 커밋 정리

현재 브랜치의 커밋들을 분석하고 정리 방안을 제안한다.

### 2.1 현재 커밋 분석

```bash
# base 브랜치(develop) 이후의 커밋들을 분석
git log develop..HEAD --oneline
```

### 2.2 정리 방안 제안

다음 관점에서 정리가 필요한지 판단하고, 필요한 경우 사용자에게 제안한다:

| 정리 유형 | 판단 기준 | 예시 |
|----------|----------|------|
| **Squash** | 같은 기능에 대한 수정/보완 커밋이 연속됨 | `feat: 댓글 등록` + `fix: 댓글 등록 오타` → 합치기 |
| **Reword** | 메시지가 컨벤션과 불일치 | `wip` → `PS-85 댓글 등록 도메인 구현` |
| **Reorder** | 논리적 순서가 맞지 않음 | 테스트가 구현보다 뒤에 커밋됨 → 순서 교정 |

**정리 방안을 사용자에게 보여준다**:
```
현재 커밋 ({N}개):
  1. abc1234 PS-85 wip 댓글 엔티티
  2. def5678 PS-85 댓글 등록 로직
  3. ghi9012 PS-85 오타 수정
  4. jkl3456 PS-85 댓글 조회 API

정리 제안:
  - 커밋 1,2,3 → Squash → "PS-85 뉴스 댓글 등록 도메인 및 서비스 구현"
  - 커밋 4 → Reword → "PS-85 뉴스 댓글 목록 조회 API 구현"

정리 후 결과 (2개 커밋):
  1. PS-85 뉴스 댓글 등록 도메인 및 서비스 구현
  2. PS-85 뉴스 댓글 목록 조회 API 구현

이대로 정리할까요? (Y/n)
```

### 2.3 정리 실행

사용자 승인 후 `git rebase -i`를 통해 정리를 실행한다. 대화형 rebase 대신 스크립트 방식으로 자동 실행한다:

```bash
# 예: squash + reword를 자동으로 수행
GIT_SEQUENCE_EDITOR="sed -i 's/pick {hash}/squash {hash}/'" git rebase -i develop
```

정리 중 충돌이 발생하면 § 4의 충돌 해결 절차를 따른다.

## 3. Base 브랜치 동기화 & Rebase

### 3.1 백업 브랜치 생성

Rebase는 히스토리를 변경하므로 반드시 백업을 먼저 생성한다:

```bash
git branch backup/{브랜치명}_{YYMMDD}
```

백업 생성을 사용자에게 안내한다:
```
백업 브랜치를 생성했습니다: backup/{브랜치명}_{YYMMDD}
문제 발생 시 이 브랜치로 복구할 수 있습니다.
```

### 3.2 Base 브랜치 최신화 확인

```bash
git fetch origin develop
git log HEAD..origin/develop --oneline
```

- **최신인 경우** (출력 없음): "develop이 최신 상태입니다." → § 3.4로 건너뛰기
- **최신이 아닌 경우**: 새로운 커밋 목록을 보여주고 rebase 진행

### 3.3 Rebase 실행

```bash
git rebase origin/develop
```

### 3.4 Rebase 완료 확인

성공 시:
```
✅ Rebase 완료. develop 최신 커밋 위에 작업 커밋이 깔끔하게 배치되었습니다.
백업 브랜치 backup/{브랜치명}_{YYMMDD}를 삭제할까요? (Y/n)
```

## 4. 충돌 해결

Rebase 또는 커밋 정리 중 충돌이 발생하면:

### 4.1 충돌 현황 표시

```
⚠️ Rebase 충돌이 발생했습니다.

충돌 파일 ({N}개):
  1. {파일 경로} — {충돌 유형: 양쪽 수정 / 삭제 후 수정 / 등}
  2. {파일 경로} — {충돌 유형}

현재 진행 중인 커밋: {커밋 메시지}
```

### 4.2 충돌 해결 가이드

각 충돌 파일에 대해:

```
### {파일 경로}

**충돌 원인**: {왜 충돌이 발생했는지 맥락 설명}
- 현재 브랜치(ours): {어떤 변경을 했는지}
- develop(theirs): {develop에서 어떤 변경이 있었는지}

**추천 해결 방법**: {어떻게 병합하는 것이 적절한지}
```

### 4.3 사용자 개입 요청

자동 해결이 불가능한 경우:

```
자동 해결이 어려운 충돌입니다. 수동 해결이 필요합니다.

다음 중 선택해주세요:
  A) VSCode에서 충돌을 직접 해결 (code {파일 경로})
  B) 현재 브랜치의 변경을 우선 적용 (ours)
  C) develop의 변경을 우선 적용 (theirs)
  D) Rebase를 중단하고 백업 브랜치로 복구

사용자가 해결 후 계속 진행하려면 알려주세요.
```

**D 선택 시 복구 절차**:
```bash
git rebase --abort
git checkout {브랜치명}
git reset --hard backup/{브랜치명}_{YYMMDD}
```
