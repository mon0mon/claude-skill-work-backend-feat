# 프로젝트 문서 관리 가이드

작업 완료 후, 프로젝트 내부에 공유 가능한 지식 문서를 생성/관리한다. 단순 작업 이력이 아닌, 도메인 지식, API 설계 근거, 에러 처리 전략 등 **추후 유지보수와 AI 맥락 이해에 도움이 되는 문서**를 목표로 한다.

## 1. 문서 분류 체계

프로젝트 CLAUDE.md의 문서 관리 규칙을 따른다:

| 경로 | 용도 | git 추적 | 기준 |
|------|------|---------|------|
| `docs/product/` | 제품 동작/구조 설명 | O | 6개월 후 유지보수 시 읽을 문서 |
| `docs/ai/` | AI 작업 산출물 | X | 실행 계획, 리뷰 결과, 체크리스트 |

**하위 타입** (필요에 따라 생성):

```
docs/product/
├── domain/          # 도메인 모델, 비즈니스 규칙, 용어 정의
├── api/             # API 설계 근거, 엔드포인트 문서, 버전 관리
├── error/           # 에러 코드 체계, 예외 처리 전략
├── architecture/    # 아키텍처 결정 기록 (ADR), 모듈 구조
└── improvement/     # 향후 개선사항, 기술 부채, 리팩토링 계획
```

**파일명 형식**: `{브랜치명}_{문서타입}_{타이틀}.md` (snake_case)
예: `PS_82_spec_ai_column_image.md`

## 2. 문서 생성 판단 기준

작업 완료 후, 다음 관점에서 문서 생성/업데이트가 필요한지 판단한다:

### 2.1 새 문서 생성이 필요한 경우

| 상황 | 문서 유형 | 예시 |
|------|----------|------|
| 새 도메인/엔티티 추가 | `domain/` | 도메인 모델의 비즈니스 규칙, 상태 전이, 불변 조건 |
| 새 API 엔드포인트 추가 | `api/` | 설계 의도, 인증/인가 방식, 요청/응답 계약 |
| 새 에러 코드/예외 추가 | `error/` | 에러 코드 체계, 클라이언트 대응 가이드 |
| 아키텍처 결정 | `architecture/` | 왜 이 방식을 선택했는지 (ADR 형식) |
| 향후 개선 가능성 발견 | `improvement/` | 현재 스코프에서 제외한 사항, 기술 부채 |

### 2.2 기존 문서 업데이트가 필요한 경우

| 상황 | 동작 |
|------|------|
| 기존 도메인에 필드/관계 추가 | 해당 도메인 문서 업데이트 |
| API 변경 (파라미터, 응답 구조) | 해당 API 문서 업데이트 |
| 에러 코드 추가/변경 | 에러 문서 업데이트 |
| 이전 개선사항이 구현됨 | improvement 문서에서 해당 항목 완료 처리 |

## 3. 문서 관리 작업

### 3.1 자동 수행 (승인 불필요)

- 새 문서 생성 (작업에서 자연스럽게 도출된 내용)
- 기존 문서의 소규모 업데이트 (필드 추가, 오타 수정, 날짜 갱신)

### 3.2 사용자 승인 필요

다음 작업은 영향 범위가 크거나 복구가 어려우므로 반드시 사용자 승인을 받는다:

| 작업 | 승인 이유 |
|------|----------|
| **문서 삭제** | 복구 불가 |
| **문서 통합** (여러 파일 → 하나) | 기존 참조 링크 깨질 수 있음 |
| **문서 분리** (큰 파일 → 여러 개) | 파일 구조 변경 |
| **문서 구조 리팩토링** (디렉토리 재구성) | 전체 구조 영향 |

승인 요청 형식:
```
다음 문서 작업을 제안합니다:

{작업 유형}: {구체적 내용}
- 대상: {파일 경로}
- 이유: {왜 이 작업이 필요한지}
- 영향: {영향받는 범위}

진행할까요? (Y/n)
```

## 4. 문서 작성 원칙

### 반드시 포함할 내용
- **왜(Why)**: 이 설계/결정을 한 이유, 배경, 제약조건
- **무엇(What)**: 구체적인 내용, 규칙, 구조
- **주의사항**: 함정, 알려진 한계, 향후 변경 가능성

### 피해야 할 내용
- 코드를 그대로 복사 (코드는 코드에서 읽으면 됨)
- 일시적인 상태 정보 (현재 진행 중인 작업, 날짜별 변경 이력)
- git log로 확인 가능한 정보 (누가 언제 변경했는지)

### 문서 간 연결
- 관련 문서 간 상호 참조 링크를 추가한다
- 스펙 문서 → 도메인 문서 → API 문서로 자연스럽게 연결되도록 한다

## 5. 워크플로우 통합

Phase 9 (머지 & 완료) 이후, 다음 순서로 문서 작업을 수행한다:

1. **기존 문서 탐색**: `docs/product/` 하위를 탐색하여 이번 작업과 관련된 기존 문서를 찾는다
2. **업데이트 대상 식별**: 변경된 도메인/API/에러 코드에 해당하는 문서가 있는지 확인
3. **문서 생성/업데이트 계획 제시**: 어떤 문서 작업을 할지 사용자에게 요약 제시
4. **승인 후 실행**: 자동 수행 가능한 작업은 바로, 승인 필요한 작업은 확인 후 진행
5. **결과 안내**: 생성/수정된 문서 목록을 보여준다
