# 코드 리뷰 체크리스트

구현 완료 후 아래의 모든 관점을 빠짐없이 체크한다. `requesting-code-review` 스킬이 사용 가능하면 함께 활용한다.

## 1. 스펙 정합성
- 스펙 문서의 모든 요구사항이 구현되었는지 확인
- 스펙에 없는 기능이 추가되지 않았는지 확인
- 스펙의 설계 결정(DD)이 있는 경우: 기각된 대안의 패턴이 구현에 혼입되지 않았는지 확인 (예: DD에서 이벤트 소싱을 기각했는데 이벤트 기반 코드가 작성된 경우)

## 2. 아키텍처 컨벤션 준수
다음 아키텍처 규칙 위반 여부를 체크한다:
- **헥사고날 아키텍처 위반**: api 모듈이 common의 repository를 직접 사용하는지, common 모듈 간 유즈케이스 호출이 있는지 (밀접하지 않은 도메인 간)
- **계층 침범**: 컨트롤러에서 비즈니스 로직 수행, 서비스에서 HTTP 관련 로직 처리
- **패키지 구조 위반**: 파일이 컨벤션에 맞지 않는 위치에 있는지
- **네이밍 컨벤션 위반**: 포트 인터페이스, 컨트롤러, DTO 등의 이름이 프로젝트 규칙을 따르는지
- **의존 방향 위반**: 내부 레이어가 외부 레이어에 의존하는지

## 3. JPA & 데이터베이스 문제
다음 패턴을 반드시 확인한다:

| 체크 항목 | 설명 | 발생 조건 |
|----------|------|----------|
| **N+1 쿼리** | 연관 엔티티를 루프에서 개별 조회 | `@OneToMany`/`@ManyToOne` 관계에서 Lazy 로딩 시 컬렉션 순회 |
| **MultiBagFetchException** | 2개 이상의 `@OneToMany` 관계를 동시에 fetch join | `List` 타입 컬렉션 2개 이상 fetch join |
| **Pagination In-Memory** | 페이징 쿼리가 모든 데이터를 메모리에 로드 후 페이징 | fetch join + Pageable 조합 시 |
| **Detached Entity Dirty Checking** | 트랜잭션 밖에서 엔티티 수정 후 dirty checking 기대 | `@Transactional` 범위 밖에서 setter 호출 |
| **LazyInitializationException** | 트랜잭션 밖에서 Lazy 로딩 프록시 접근 | 컨트롤러/뷰 레이어에서 미로딩 연관 엔티티 접근 |
| **영속성 컨텍스트 혼동** | 같은 엔티티의 다른 인스턴스 공존 | 서로 다른 트랜잭션/영속성 컨텍스트에서 동일 ID 엔티티 조작 |
| **CascadeType 오용** | 의도치 않은 연쇄 삭제/저장 | `CascadeType.ALL`이나 `REMOVE` 사용 시 |
| **orphanRemoval 부작용** | 컬렉션 교체 시 의도치 않은 삭제 | `orphanRemoval = true`와 컬렉션 재할당 |

## 4. 동시성 & 트랜잭션 문제

| 체크 항목 | 설명 | 발생 조건 |
|----------|------|----------|
| **Lost Update** | 동시 수정 시 나중 트랜잭션이 앞선 변경 덮어씀 | 락 없이 read-modify-write 패턴 |
| **Race Condition** | 동시 요청으로 데이터 불일치 | 중복 등록, 동시 좋아요/취소, 잔액 차감 등 |
| **Deadlock** | 두 트랜잭션이 서로의 락 대기 | 여러 테이블을 다른 순서로 락 획득 |
| **잘못된 락 사용** | 불필요한 비관적 락으로 성능 저하 | `@Lock(PESSIMISTIC_WRITE)` 과다 사용 |
| **트랜잭션 범위 과대** | 긴 트랜잭션이 DB 커넥션 점유 | 외부 API 호출을 트랜잭션 안에서 수행 |
| **비동기 내 동기 블로킹** | 비동기 컨텍스트에서 동기 코드 실행 | `@Async` 메서드 안에서 동기적 DB 조회 반복 |

## 5. 성능 문제

| 체크 항목 | 설명 | 발생 조건 |
|----------|------|----------|
| **대량 데이터 풀스캔** | 인덱스 없이 전체 테이블 스캔 | WHERE 절에 인덱스 미적용 컬럼 사용 |
| **불필요한 전체 컬럼 조회** | SELECT * 패턴 | 엔티티 전체를 조회하나 일부 필드만 사용 |
| **메모리 누수 패턴** | 큰 컬렉션을 메모리에 한번에 로드 | `findAll()` 후 스트림 필터링 |
| **커넥션 풀 고갈** | 장시간 커넥션 점유 | 트랜잭션 안에서 외부 API 호출, 파일 I/O |
| **캐시 미활용** | 반복 조회되는 데이터에 캐시 미적용 | 자주 조회되나 변경 빈도 낮은 데이터 |
| **비동기 오버헤드** | 단순 작업에 비동기 처리 | 작은 작업에 `@Async` 적용으로 스레드 전환 비용 발생 |

## 6. 일반적인 백엔드 실수

| 체크 항목 | 설명 |
|----------|------|
| **@Transactional(readOnly=true) 누락** | 읽기 전용 메서드에 readOnly 미설정 |
| **예외 처리 누락** | 외부 API 호출, 파일 I/O 등에서 예외 미처리 |
| **입력 검증 누락** | `@field:` 접두어 없는 Bean Validation, 서비스 레이어 검증 누락 |
| **Enum 매핑 오류** | `@Enumerated(EnumType.ORDINAL)` 사용 (STRING이어야 함) |
| **FetchType.EAGER 사용** | `@ManyToOne`/`@OneToMany`에서 EAGER 로딩 |
| **equals/hashCode 미구현** | 엔티티 비교 시 ID 기반 비교 미구현 |
| **날짜/시간 처리 오류** | 타임존 미고려, `LocalDateTime` vs `ZonedDateTime` 혼용 |
| **민감 정보 노출** | 응답 DTO에 비밀번호, 내부 ID 등 노출 |
| **findByIdOrThrow 미사용** | `findById().orElseThrow()` 대신 프로젝트 확장 함수 미사용 |

## 7. 테스트 커버리지
- 실행 계획의 모든 테스트가 작성되었는지
- 테스트가 의미 있는 검증을 수행하는지 (단순 not null 체크가 아닌)
- 테스트 픽스처가 `InstancioUtil.kt`에 추가되었는지
