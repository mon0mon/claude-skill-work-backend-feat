# 서브에이전트 모델 배정 가이드

이 워크플로우의 각 Phase/Step은 작업 성격에 따라 적절한 모델을 사용하는 서브에이전트로 위임할 수 있다. 토큰 비용을 절약하고 실행 시간을 단축하기 위해, 무거운 작업은 opus, 중간 작업은 sonnet, 가벼운 작업은 haiku로 분배한다.

## 모델 선택 기준

| 모델 | 적합한 작업 | 핵심 강점 |
|------|-----------|----------|
| **opus** | 아키텍처 결정, 트레이드오프 분석, 스펙 작성, 복잡한 리팩토링, 문서 설계 | 깊은 추론, 복잡한 판단 |
| **sonnet** | 코드 생성/수정, 코드 리뷰, 탐색, 디버깅, 테스트, 구현 | 빠른 코드 생성, 균형잡힌 성능 |
| **haiku** | 단순 탐색, 형식 변환, 데이터 추출, 로그 분석, 파일 복사 | 속도, 저비용 |

## 서브에이전트 위임 규칙

### 위임 가능한 경우
- 사용자 승인이 필요하지 않은 독립적인 작업
- 입출력이 명확하고 다른 Step에 의존하지 않는 작업
- 연속된 여러 Step을 하나의 서브에이전트에 묶어 실행 가능

### 위임하면 안 되는 경우
- 사용자와 직접 대화해야 하는 경우 (승인, 선택, 질문)
- 이전 Step의 결과에 따라 분기가 필요한 경우
- 메인 에이전트의 컨텍스트가 필요한 경우

### 병렬 실행 가능한 조합
다음 Step들은 서로 의존하지 않으므로 병렬 서브에이전트로 실행할 수 있다:
- Phase 5: Step 5.2 (Jira 티켓) + Step 5.4 (브랜치 생성) — Jira 티켓 키가 필요하므로 실제로는 순차
- Phase 6: Step 6.1 (테스트 작성) 내에서 독립된 도메인 테스트들을 병렬 생성
- Phase 9: Step 9.3 (워크트리 동기화) + Step 9.4 (브랜치 정리) + Step 9.5 (Jira 완료) — 병렬 가능
